Modularity Review

Model: Balanced Coupling (Vlad Khononov) Scope: Entire codebase Date: 2026-04-16

Follow-up (2026-04-17)

Findings §1 (TopicPublisherPair public API) and §3 (poller-backed @AutoConfiguration ordering) are implemented in the current codebase. The analysis below is kept as written for context.

Executive Summary

The module boundaries in spring-crablet are sound. Public APIs are stable, internal implementations are well-hidden behind factory methods and auto-configurations, and the metrics layer is a textbook inversion-of-control design. One integration was concretely unbalanced (TopicPublisherPair leaking via OutboxManagementService; see §1 — since fixed). Everything else is either well-balanced or tolerable given the project's solo-developer context.

Integration Analysis

✓ Resolved

1. OutboxManagementService leaked an internal type (was: Unbalanced — Fix)

OutboxAutoConfiguration was using com.crablet.outbox.internal.TopicPublisherPair as the type parameter for every infrastructure bean, forcing consumers (e.g. wallet-example-app) to import from a *.internal package. A public com.crablet.outbox.TopicPublisherPair existed but was unused.

DimensionAssessment
StrengthModel — internal type in every public method signature
DistanceCross-module — escapes the library; consumers must import internals
VolatilityLow — "identify a processor by (topic, publisher)" is stable
BALANCE = (HIGH_STRENGTH XOR HIGH_DISTANCE) OR NOT LOW_VOLATILITY = FALSE OR TRUE = TRUE

Low volatility technically balanced the equation, but the XOR leg was false — both strength and distance were high. Tolerable, but an API design error.

Resolution. com.crablet.outbox.internal.TopicPublisherPair was removed. All infrastructure and consumers now use com.crablet.outbox.TopicPublisherPair with compact-constructor validation.

✓ Resolved

2. CommandWebAutoConfiguration imported command.internal.DiscoveredCommandRegistry (was: Acceptable)

CommandWebAutoConfiguration (in crablet-commands-web) creates a DiscoveredCommandRegistry by calling the static factory. The class now lives in the public package: 

// CommandWebAutoConfiguration.java
import com.crablet.command.DiscoveredCommandRegistry;  // now public

@Bean
public DiscoveredCommandRegistry discoveredCommandRegistry(List<CommandHandler<?>> handlers) {
    return DiscoveredCommandRegistry.fromHandlers(handlers);
}
DimensionAssessment
StrengthFunctional — creates and passes the registry; no business logic shared
DistanceAdjacent — declared pom.xml dependency; same module family
VolatilityLow — handler discovery mechanics are stable
BALANCE = (FUNCTIONAL XOR LOW_DISTANCE) OR NOT LOW = TRUE OR TRUE = TRUE

Balanced. Resolution. DiscoveredCommandRegistry promoted from com.crablet.command.internal to com.crablet.command.

✓ Resolved

3. Poller-backed modules used @Configuration instead of @AutoConfiguration (was: Correctness risk — Improve)

ViewsAutoConfiguration, OutboxAutoConfiguration, and AutomationsAutoConfiguration were annotated @Configuration, preventing them from declaring after = EventPollerAutoConfiguration.class. Ordering worked at runtime via bean dependency resolution, but there was no explicit contract — a future Spring Boot version could have changed resolution order.

Resolution. All three now use @AutoConfiguration(after = EventPollerAutoConfiguration.class) while remaining registered in META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports.

ℹ Note — Tolerable for solo, revisit if project grows

4. shared-examples-domain ↔ framework test scope (build cycle)

Framework modules use shared-examples-domain in test scope to run realistic domain tests. shared-examples-domain depends on crablet-eventstore and crablet-commands to compile. The resulting cycle requires stub JARs and a specific Makefile build order.

DimensionAssessment
StrengthModel — tests instantiate real wallet types; a CommandHandler API change propagates to shared-examples-domain
DistanceSame repo, same developer — low effective distance
VolatilityMedium — framework public API is actively evolving

Model strength + low distance: the XOR leg is false (both medium), but low distance saves the balance. The design is correct for a solo project. It becomes problematic if a second contributor joins and owns only one module, or if the example domain is extracted as a separate repository — at that point, distance increases and the build cycle becomes an unbalanced coupling.

What Is Working Well

Pub/sub metrics — exemplary design

crablet-metrics-micrometer has zero knowledge of how events are generated. It subscribes via @EventListener to metric events published by each module via ApplicationEventPublisher. Strength: contract. Distance: cross-module. Volatility: very low. Perfectly balanced.

Factory abstractions over implementations

CommandExecutors.create(...) and EventProcessorFactory.createProcessor(...) mean that CommandExecutorImpl and EventProcessorImpl are never referenced by name outside their own modules. The internal package boundary is real, not just a naming convention.

Optional dependency done right

AutomationsAutoConfiguration uses ObjectProvider<CommandExecutor> with a clear startup failure message when in-process handlers are declared without a CommandExecutor bean. The optional coupling is explicit, validated early, and documented in the error message.

Activation by inclusion

crablet-commands-web is activated by adding it to pom.xml and declaring one required bean (CommandApiExposedCommands). If absent, Spring Boot fails at startup with a clear missing-bean message — no silent misconfiguration. This is a cleaner opt-in than the @ConditionalOnProperty pattern used by the poller modules.
