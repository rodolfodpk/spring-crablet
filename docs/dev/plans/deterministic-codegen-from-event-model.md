# Deterministic Codegen From Event Model

## Goal

Move Crablet code generation to this default architecture:

```text
AI-assisted:
  conversation -> event-model.yaml

Deterministic:
  event-model.yaml -> Java source, migrations, k8s, scenario scaffolds
```

The generator should not require an LLM API key to produce application code. AI remains useful for
event modeling, YAML authoring, review, explanation, and optional repair suggestions, but generated
source should be repeatable from `event-model.yaml`.

## Why

The current LLM-backed Java generation is feasible but hard to trust as a product default:

- a Crablet user may have Claude Code or Codex access but no Anthropic/OpenAI API key
- identical inputs can produce different Java output
- CI cannot rely on live model calls
- snapshot tests only prove one generated result, not the generation workflow
- prompt drift and provider differences can break code without a code change

Deterministic generation makes the model file the contract:

```text
same event-model.yaml + same generator version = same generated files
```

## User Experience

Primary workflow:

```bash
crablet plan
crablet generate
./mvnw verify
```

or through MCP:

```text
crablet_plan
crablet_generate
```

No LLM provider configuration should be required for `plan`, `generate`, `k8s`, or
`sync-scenarios`.

AI-frontends such as Claude Code, Cursor, and Codex should help users create and refine
`event-model.yaml`, then call deterministic tools.

## File Ownership Model

Use ownership boundaries instead of attempting to merge arbitrary user edits into regenerated files.

### Machine-owned files

These may be overwritten on every generation run:

- domain event sealed interfaces and records
- command records
- generated handler contracts
- state records
- state projectors (once explicit state modeling exists in the YAML schema; user-owned until then)
- query pattern helpers
- view projectors
- view row contracts, if introduced
- generated Kubernetes manifests
- generated metadata files

Generated files live in current user-facing packages and carry a generated header:

```java
// Generated by Crablet from event-model.yaml. Do not edit.
```

### User-owned files

These are created only when missing and never overwritten:

- command handler implementations
- automation implementations
- outbox publisher implementations
- custom controllers and services
- scenario tests after initial scaffold
- hand-written migrations
- any file outside generated ownership roots

Example shape:

```text
machine-owned (generated header, tracked in manifest):
  com.example.loan.domain.LoanApplicationSubmitted
  com.example.loan.command.SubmitLoanApplication
  com.example.loan.command.SubmitLoanApplicationCommandHandler

user-owned (create-only, never overwritten):
  com.example.loan.command.SubmitLoanApplicationHandler
```

Generated contract:

```java
public interface SubmitLoanApplicationCommandHandler
        extends IdempotentCommandHandler<SubmitLoanApplication> {
}
```

User implementation:

```java
@Component
class SubmitLoanApplicationHandler implements SubmitLoanApplicationCommandHandler {
    // business decision code
}
```

## Migration Policy

Flyway migrations need stricter rules than Java files:

- never edit an existing migration once generated
- generate a new migration when the event model changes a view table
- include stable versioning or timestamping in generated migration names
- future: detect drift between the model and existing migrations
- future: fail with an actionable message when a destructive schema change needs a human decision

Initial generated migrations can be machine-owned while they are new. After they exist on disk, treat
them as append-only historical artifacts.

## Generator Design

Replace LLM agents that produce Java with deterministic generators:

- `EventsGenerator`
- `CommandsGenerator`
- `ViewsGenerator`
- `AutomationsGenerator`
- `OutboxGenerator`
- `ScenarioScaffoldGenerator`
- `K8sGenerator` already follows this direction

Use structured rendering, not stringly ad hoc output where avoidable:

- JavaPoet or equivalent for Java source
- a small SQL template layer for migrations
- existing YAML/object rendering for k8s

Keep `ArtifactPlanner` as the preview of what deterministic generation will produce.

## Regeneration Manifest

Add a generated manifest to track ownership and generator version:

```text
.crablet/generated-manifest.json
```

Suggested fields:

- generator version
- source `event-model.yaml` hash
- generated file list
- ownership kind
- template/generator id
- last generated timestamp

Use the manifest to:

- delete stale generated files when model entries are removed
- refuse to overwrite unknown user files
- show meaningful diffs in `plan`
- support `sync` and drift checks

`crablet repair` must either update the manifest when it writes machine-owned files, or be
explicitly documented as manifest-unaware. Silent writes to machine-owned files without manifest
updates create drift that `generate` cannot detect.

## AI Role After The Change

Keep AI optional and explicit:

- event-modeling workshop: conversation -> `event-model.yaml`
- `crablet explain`: explain the generated plan
- `crablet suggest`: propose model changes based on compile/test errors
- `crablet repair`: optional, never part of default CI

Do not make `generate` depend on a live model call.

## Phased Plan

### Phase 1: Freeze The Contract

Decisions to record in docs before any generator work begins:

- use current user-facing packages plus a generated file header and manifest entry (not `.generated` sub-packages)
- generated files are committed to application repos by default; CI verifies they match a regeneration run rather than running codegen as a build step
- view migrations use Flyway timestamp-based versioning (`V20260525HHMMSS__...`) to avoid merge conflicts across feature branches
- stale machine-owned file deletion requires an explicit `--prune` flag; `generate` warns about stale files but does not remove them automatically
- **interim decision:** keep `StateProjector` user-owned (create-only) until `event-model.yaml` gains explicit state derivation metadata (`state:` section or per-event `contributes:` annotations); this unblocks Phase 2 immediately — deterministic `StateProjector` generation is deferred to a later phase when the schema can support it

Implementation tasks:

- document generated vs user-owned files
- update template docs and AI workflow docs
- add generated headers to machine-owned files
- add tests that assert scenario scaffolds are create-only
- `make codegen-snapshot-verify` already exists; add assertions that cover the ownership boundary

Acceptance:

- all five decisions above are recorded in docs and `crablet-codegen/CLAUDE.md`
- docs clearly tell users what to edit
- `make codegen-snapshot-verify` stays green

### Phase 2: Deterministic Events And Commands

- implement deterministic event sealed interface and record generation
- implement deterministic command record generation with validation
- implement deterministic handler contract generation
- stop using LLM for events and command contracts
- keep user handler implementations create-only

Acceptance:

- events, command records, and handler contracts for the loan model generate without an LLM key
- generated files match the snapshot or an intentional updated snapshot
- codegen tests cover naming, packages, tags, field types, and validation constraints

### Phase 3: Deterministic Projectors And Views

- implement create-only user-owned `StateProjector<T>` scaffolding; deterministic regeneration is deferred until `event-model.yaml` gains explicit state derivation metadata
- implement deterministic query helper generation
- implement deterministic view projector generation
- implement deterministic view migration generation with append-only, timestamp-versioned policy

Acceptance:

- loan view projector and migration compile from deterministic generation
- a new migration is generated when the view schema changes; existing migrations are never modified

Note: migration drift detection (detecting when generated migrations diverge from the current model
and failing on destructive schema changes) is a non-trivial feature and should be planned and scoped
separately rather than treated as a Phase 3 acceptance criterion.

### Phase 4: Deterministic Automations And Outbox

- generate automation contracts and scaffolds deterministically
- generate outbox contracts and scaffolds deterministically
- keep external integration code user-owned

Acceptance:

- course or wallet workflow with automations/outbox generates without an LLM
- user implementation files are never overwritten

### Phase 5: Remove LLM From Default Generate

- rename current LLM path to an explicit experimental command, if still useful
- make `generate` deterministic
- make provider configuration unnecessary for normal app generation
- keep MCP `crablet_generate` mapped to deterministic generation

Acceptance:

- clean template app can run `make generate && make verify` with no LLM API key
- CI can run deterministic codegen snapshot checks offline

## Decisions

These were open questions; answers are now recorded here and must be reflected in Phase 1 docs.

**Package naming:** Use current user-facing packages with a `// Generated by Crablet` header and
manifest entry. Do not use `.generated` sub-packages — they leak implementation detail into user
code and make imports awkward.

**Generated files in app repos:** Commit generated files by default. CI verifies they match a
regeneration run; it does not run codegen as a mandatory build step. This keeps builds fast and
offline-capable.

**Migration versioning:** Use Flyway timestamp-based names in UTC: `VyyyyMMddHHmmss__create_table.sql`.
If multiple migrations are generated within the same second, append a numeric suffix (`_2`, `_3`).
Sequential numbers (`V100`, `V101`) cause merge conflicts when multiple branches each add a view.

**Stale file deletion:** Require an explicit `--prune` flag. The `generate` command warns about
stale machine-owned files but never removes them automatically. Automatic deletion is unsafe on
feature branches where a model entry may be temporarily absent.

**StateProjector generation strategy:** Deterministic `StateProjector` generation is not feasible
from the current YAML — the generator cannot infer which state flags and accumulated fields to
emit from event presence alone. The path forward requires one of:

- a `state:` section in `event-model.yaml` listing fields and their derivation rules
- per-event `contributes:` metadata mapping event fields to state fields
- treating `StateProjector` as user-owned (create-only) until the YAML schema supports it

Interim decision (Phase 1): keep `StateProjector` user-owned until the YAML schema supports
explicit state derivation. This unblocks Phase 2 — events, command records, and handler contracts
can proceed without it. If a subclass pattern is adopted later, the generated base must **not**
be `@Component` — Spring bean selection becomes ambiguous if both the base and a subclass are
visible to the container. The user-owned subclass or a factory carries the `@Component` annotation.

## Near-Term Recommendation

Stop investing in prompt-only fixes as the main path. Keep the fence-stripping and repair loop as
defensive tooling, but do not extend it. Shift effort to deterministic generators for events,
commands, projectors, views, and migrations.

Start `EventsGenerator` and command record/contract generation after Phase 1 ownership decisions
are recorded. `AutomationsGenerator` and `OutboxGenerator` can follow in the same window — they
are purely structural and have no design ambiguity. `StateProjector` deterministic generation is deferred — Phase 1 records the interim decision to
keep it user-owned, which is sufficient to start Phase 2.
